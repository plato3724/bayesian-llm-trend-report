# 项目简版说明

更新日期：2026-04-13

这份文档面向产品视角和日常使用者，目标是用最短路径说明：这个项目现在能做什么、你平时怎么用、系统在什么情况下会停下来等你确认。

## 1. 这个项目现在是什么

这是一个“文章驱动的趋势验证系统”，不是普通的资讯收集器。

它做的事情是：

1. 你把文章链接发进 GitHub issue
2. 系统自动抓取全文
3. 系统自动提取文章里的结构化观点和事实
4. 系统自动核实这些内容
5. 只有被核实、并且确实支持某个趋势假设的证据，才会更新报告
6. 如果文章发现了现有框架解释不了的新模式，系统会生成“候选假设”供你决定是否升格

所以它的核心不是“收集文章”，而是“把文章变成趋势证据”。

## 2. 现在已经能做到什么

### 自动主链路

现在已经可以通过 GitHub Actions 完成这条链路：

- issue 加文章链接
- 自动 ingest
- 自动 draft claims
- 自动 draft verification
- 自动更新报告

如果 verification 过程没有触发风险，系统会自动 apply。

### GitHub 内人工确认

如果系统认为某篇文章需要人工确认，不需要你本地改 JSON 文件，直接在 GitHub issue 评论里回复命令即可：

- `/approve`
- `/approve-safe`
- `/reject`

这意味着 held review 已经在 GitHub 内闭环。

### 趋势假设可读性增强

现在报告里的趋势卡片已经不是只看一个概率数字了。

你可以：

- 展开看支撑这个趋势的具体证据
- 跳到对应文章查看证据来源
- 看每个核心假设的详细理论页
- 看到“本次新增文章”推动了哪张趋势卡片

### 工具索引自动汇总

如果新文章中介绍了新的工具/项目，系统会在核实后自动把它补进首页工具索引，而不需要你手工再维护一遍。

### 候选假设机制

当文章里出现“已经核实，但现有趋势假设解释不了”的证据时，系统会把它们收集成候选假设。

你现在可以：

- 在报告里看到候选假设
- 看到它的 `candidate_id`
- 在 GitHub issue 评论里决定：
  - `/promote-candidate <candidate_id>`
  - `/reject-candidate <candidate_id>`

## 3. 你平时怎么使用这个系统

最常见的使用方式只有两种。

### 场景 A：上传新文章

你只需要：

1. 去 GitHub 仓库新建一个 issue
2. 在 issue 里放文章链接
3. 给它加上 `article` 标签

之后系统会自动开始处理。

你主要看两个地方：

- GitHub Actions 是否成功跑完
- 报告页有没有出现新的文章状态、趋势变化或候选假设

### 场景 B：系统要求你人工判断

如果系统没有自动 apply，它会在 issue 里提示你当前文章被 held。

这时你只需要回复：

- `/approve`
  - 直接人工通过并 apply
- `/approve-safe`
  - 保留事实，但不让这篇文章推动趋势 posterior
- `/reject`
  - 不应用这次结果

如果系统生成了候选假设，你只需要回复：

- `/promote-candidate <candidate_id>`
  - 把候选升格成正式趋势假设
- `/reject-candidate <candidate_id>`
  - 拒绝这个候选

## 4. 你现在看到的页面应该怎么理解

### 核心趋势

这是当前系统认为最值得持续跟踪的正式趋势假设。

每张卡代表：

- 一个正式 hypothesis
- 当前 posterior 概率
- 支撑它的证据
- 对应的详细理论说明

### 候选假设

这是“框架可能需要生长”的信号层。

候选假设说明：

- 有些文章里的证据已经核实过
- 但它们还不适合挂到现有正式趋势假设上
- 系统把这类证据收集起来，形成候选假设
- 它们不会自动进入 posterior，必须由你决定是否 promote

### 工具索引

这里汇总项目当前识别到的重要工具/项目。

它既包括你手工维护的条目，也包括系统根据新文章自动补充的条目。

### 待处理 / Held / 排除

这些状态不是报错，而是流程状态。

- `待处理`
  - 说明文章还在链路里，还没走完
- `held`
  - 说明系统不确定，暂停等待你确认
- `排除`
  - 说明文章可能被核实了，但当前不足以推动任何正式趋势更新

## 5. 当前系统还没有做到什么

下面这些目前还没有完全做好，属于后续优化方向。

### 候选假设提醒还不够主动

现在系统已经能生成候选，也能在 GitHub 里 promote/reject。

但它还不会主动很强地提醒你：

- 哪个 candidate 最值得 review
- 推荐你 promote 还是 reject
- 直接给你可复制命令

这是下一阶段最值得补的体验优化。

### 候选假设质量还不够稳定

现在 candidate 更像“已核实未映射证据的聚类结果”。

它已经可用，但还不是高质量的理论化产物。后续还需要增强：

- 命名
- 聚类稳定性
- rationale 的可读性
- promote 的门槛控制

### 正式 hypothesis 仍然不会自动生长

系统现在能生成 candidate，但不会自动升格成正式 hypothesis。

这不是 bug，而是当前的风险控制选择。因为一旦正式 hypothesis 自动进入 posterior，就会直接影响趋势判断。

## 6. 当前最适合继续优化的方向

如果从产品和使用体验角度看，接下来最值得做的是：

1. 候选假设推荐与提醒机制
2. 候选假设质量增强
3. 候选详情页
4. 状态面板和可观察性增强

## 7. 一句话总结

这个项目现在已经从“文章处理脚本”变成了“可持续运行的趋势验证系统”。

当前最重要的变化不是主链路能不能跑通，而是：

- 它已经能自动吸收文章
- 能在 GitHub 内完成关键人工确认
- 能把新证据映射到趋势
- 也开始具备“长出新假设”的能力

后续优化的重点，是把这套系统做得更清晰、更顺手、更容易持续维护。
